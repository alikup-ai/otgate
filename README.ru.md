# otgate

[![CI](https://github.com/alikup-ai/otgate/actions/workflows/ci.yml/badge.svg)](https://github.com/alikup-ai/otgate/actions/workflows/ci.yml)

OT-aware шлюз авторизации между LLM-агентом и промышленным OPC UA сервером. Применяет к каждому обращению агента политику допуска, понимающую семантику процесса — диапазоны значений, скорости изменения, блокировки.

[English version →](README.md)

<p align="center">
  <img src="assets/demo.svg" alt="демо otgate: записи LLM-агента в реактор, отфильтрованные политикой — ALLOW, ASK, DENY с причинами" width="100%">
</p>

```
[MCP-клиент / агент] ---MCP---> [otgate] ---OPC UA---> [OPC UA сервер]
                                    |
                     [движок политик + аудит + shadow mode]
```

otgate сам является MCP-сервером. Агент подключается к нему как к обычному OPC UA MCP, а внутри otgate держит клиент к настоящему OPC UA-серверу. Для агента это прозрачно; otgate перехватывает каждый вызов, применяет политику и пишет строку в аудит.

## Зачем

- Существующие OPC UA MCP-серверы пишут в теги **без какого-либо слоя допуска** — агент имеет полный неограниченный доступ.
- Существующие guardrails для LLM (LlamaFirewall, NeMo Guardrails) работают на уровне **текста/промптов** и не понимают семантику процесса — они ничего не знают про диапазоны, скорости изменения и блокировки.
- otgate закрывает ровно этот разрыв: **OT-семантика + политика допуска + аудит.**

Это не «можно писать в тег», а «можно писать *в диапазоне X, не быстрее Y за интервал Z, и только пока аварийный тег W не активен*».

## Установка


## Пример политики

Политика — это YAML-список правил по тегам. Любой тег, **не** описанный явно, запрещён по умолчанию.

```yaml
# examples/reactor_policy.yaml
- tag: "ns=2;s=Reactor.TIC101.PV"     # температура, только чтение
  access: read

- tag: "ns=2;s=Reactor.TIC101.SP"     # уставка температуры
  access: write_with_approval          # запись требует подтверждения (ASK)
  value_range: [40, 80]                # только в диапазоне 40..80 °C
  max_rate: 5                          # не быстрее 5 °C ...
  rate_interval: 60                    # ... за 60 секунд
  interlocks:
    - tag: "ns=2;s=Reactor.ESD"        # заблокировано, пока активна ...
      condition: "== true"             # ... аварийная остановка
      action: deny
```

Уровни доступа: `read`, `write`, `write_with_approval`, `deny`.

Порядок проверок для записи (побеждает первая сработавшая): тег есть в политике → доступ → диапазон значений → блокировки → скорость изменения → подтверждение → разрешение.

## Запуск как MCP-сервер

otgate предоставляет: `read_tag`, `write_tag`, `browse`, `get_audit_log`, а также
операторские инструменты подтверждения `list_pending`, `approve`, `deny`.

```bash
otgate                      # консольный скрипт (транспорт stdio)
python -m otgate.server     # эквивалент
```

Конфигурация — через переменные окружения (чтобы бэкенд переключался без изменения кода):

| Переменная | По умолчанию | Значение |
| --- | --- | --- |
| `OTGATE_BACKEND` | `fake` | `fake` или `asyncua` |
| `OTGATE_POLICY` | `examples/reactor_policy.yaml` | путь к YAML политики |
| `OTGATE_AUDIT` | `audit.jsonl` | путь к аудит-логу |
| `OTGATE_SHADOW` | *(выкл.)* | `1`/`true` — включить shadow mode |
| `OTGATE_RATE_HISTORY` | *(in-memory)* | путь к JSONL-файлу, чтобы история скорости изменения **переживала перезапуск** |
| `OTGATE_OPCUA_ENDPOINT` | — | обязательна для бэкенда `asyncua` |
| `OTGATE_AGENT_TOKEN` | — | bearer-токен для одиночного agent HTTP-канала |
| `OTGATE_AGENTS` | — | путь к `agents.yaml` для мульти-агентного режима (переопределяет одиночный) |
| `OTGATE_OPERATOR_TOKEN` | — | bearer-токен для operator HTTP-канала (должен отличаться от всех agent-токенов) |
| `OTGATE_HOST` / `OTGATE_AGENT_PORT` / `OTGATE_OPERATOR_PORT` | `127.0.0.1` / `8770` / `8771` | host и порты HTTP |

Пример конфига MCP для Claude Desktop / Claude Code (FakeBackend):

```json
{
  "mcpServers": {
    "otgate": {
      "command": "otgate",
      "env": { "OTGATE_BACKEND": "fake" }
    }
  }
}
```

### Подключение реального OPC UA сервера

Без изменения кода — поставьте extra и укажите свой сервер в конфиге:

```bash
pip install -e ".[opcua]"
export OTGATE_BACKEND=asyncua
export OTGATE_OPCUA_ENDPOINT=opc.tcp://localhost:4840
otgate
```

Бэкенд спрятан за абстрактным интерфейсом (`otgate/backends/base.py`) с двумя реализациями — `FakeBackend` (in-memory симулятор без зависимостей) и `AsyncuaBackend` (реальный OPC UA). `asyncua` импортируется лениво, поэтому fake-бэкенд, тесты и демо работают без него.

## Shadow mode

При `OTGATE_SHADOW=1` запись, которую движок **разрешил**, **не** исполняется на бэкенде — вместо этого в аудит пишется строка «WOULD execute» с `executed: false`. Это позволяет прогнать агента и посмотреть, что бы он *натворил* с процессом, без реального воздействия.

## Workflow подтверждения человеком

Тег `write_with_approval` возвращает `ASK`. Вместо того чтобы молча отбросить
запись, otgate **паркует** её как pending-запрос с TTL и отдаёт агенту id
подтверждения. Человек затем разрешает его через операторские инструменты:

- `list_pending` — показать припаркованные запросы на запись.
- `approve <id>` — подтвердить **и** исполнить, *после повторной проверки
  политики* по текущему состоянию процесса. Если за время ожидания сработала
  блокировка или значение вышло из диапазона — подтверждение фиксируется, но
  запись **блокируется**: подтверждение — это намерение, а не слепое исполнение.
- `deny <id>` — отклонить; запись не выполнится никогда.

Запросы, на которые никто не ответил в пределах TTL, **истекают** (fail-safe:
молчание = «нет»). Каждый шаг — ASK, подтверждение/отказ, итоговое исполнение —
пишется в аудит.

> stdio-сервер выставляет все инструменты в одном процессе (годится для локальной
> работы). Для реального развёртывания запускайте **два аутентифицированных
> HTTP-канала** (ниже), чтобы агент физически не мог дотянуться до `approve`/`deny`.

## Развёртывание и безопасность

Для чего-либо серьёзнее локальных экспериментов запускайте otgate по HTTP как
**два отдельных канала**, каждый за своим bearer-токеном:

- **agent-канал** — `read_tag`, `write_tag`, `browse`
- **operator-канал** — `get_audit_log`, `list_pending`, `approve`, `deny`

Агент держит только agent-токен, поэтому даже не видит инструментов
подтверждения — изоляция по каналу, а не по договорённости.

```bash
export OTGATE_AGENT_TOKEN=$(openssl rand -hex 32)
export OTGATE_OPERATOR_TOKEN=$(openssl rand -hex 32)   # должен отличаться от agent
export OTGATE_BACKEND=asyncua
export OTGATE_OPCUA_ENDPOINT=opc.tcp://127.0.0.1:4840
otgate-http     # agent на :8770/mcp, operator на :8771/mcp
```

Каждый запрос обязан нести `Authorization: Bearer <token>`; всё остальное
получает `401`. Токены сравниваются в постоянное время. otgate откажется
стартовать, если любой из токенов не задан или если они совпадают.

**Сетевая изоляция обязательна.** otgate — это soft-слой политики: если до OPC UA
сервера можно достучаться напрямую, агент обойдёт шлюз и политика ничего не
значит. Безопасное развёртывание требует:

1. OPC UA сервер принимает подключения **только** от хоста/процесса otgate
   (loopback или приватный сегмент; всё остальное — под firewall).
2. Агент может достучаться **только** до agent HTTP-канала — не до OPC UA сервера
   и не до operator-канала.
3. Operator-канал доступен только человеку/консоли, которая подтверждает.

Иначе говоря: otgate эффективен ровно тогда, когда он *единственный* путь к
серверу. Он дополняет — но не заменяет — аппаратные блокировки / SIS.

### Несколько агентов с разными правами

Чтобы запустить несколько агентов — каждый со своим токеном, своей политикой и
своей идентичностью в аудите — укажите `OTGATE_AGENTS` на `agents.yaml`:

```yaml
# examples/agents.yaml
- id: diagnostics
  token: "…секрет A…"
  policy: policies/diagnostics.yaml   # только чтение
- id: optimizer
  token: "…секрет B…"
  policy: policies/optimizer.yaml     # может писать уставку (с подтверждением)
```

```bash
export OTGATE_AGENTS=examples/agents.yaml
export OTGATE_OPERATOR_TOKEN=$(openssl rand -hex 32)   # должен отличаться от всех agent-токенов
export OTGATE_OPERATOR_PORT=8779                       # вне диапазона agent-портов
otgate-http
```

Каждый агент обслуживается на своём порту (от `OTGATE_AGENT_PORT` и выше) за своим
токеном, поэтому агент диагностики физически не может воспользоваться правами
оптимизатора и наоборот. Все агенты делят один backend, один аудит, одну
rate-историю и один approval-store — так что rate-лимиты глобальны по тегу, а
оператор видит pending-запросы всех агентов. Каждая строка аудита несёт `agent`,
сделавшего вызов; когда оператор подтверждает припаркованную запись, она
атрибутируется агенту, который её *запросил*, а не оператору.

Старт fail-closed: otgate откажется запускаться при дублирующихся id/токенах,
отсутствующем файле политики, operator-токене, совпадающем с agent-токеном, или
operator-порте, пересекающемся с диапазоном agent-портов.

## Отказоустойчивость и health

otgate fail-**closed** при недоступности бэкенда — он никогда не превращает
сбой в молчаливый успех:

- Чтение или запись во время сбоя возвращают решение **ERROR** (не исключение,
  не поддельный успех) и пишутся в аудит с `executed: false`.
- Запись, у которой **не удаётся прочитать interlock**, **блокируется** — otgate
  не выполнит запись, которую не смог проверить по безопасности.
- Припаркованная запись, **подтверждённая** во время сбоя, не исполняется; она
  остаётся pending (или отклоняется на fail-closed повторной проверке), чтобы
  оператор мог повторить.

Реальный OPC UA бэкенд **авто-переподключается**: обрыв соединения запускает
ограниченный retry-with-backoff и один повтор операции, прежде чем сдаться с
`BackendError`. Отсутствующий тег остаётся обычной ошибкой и не вызывает
переподключений.

Каждый HTTP-канал предоставляет неаутентифицированный **`GET /health`** для
проб живости — только статус, никаких данных тегов:

```bash
curl http://127.0.0.1:8770/health
# {"status":"ok","backend":"up","pending_approvals":0}          # 200
# {"status":"degraded","backend":"down","pending_approvals":0}  # 503 при недоступном бэкенде
```

Открытие `/health` не открывает ничего другого — `/mcp` по-прежнему требует токен.

Если умирает **сам процесс** otgate, агент просто теряет свой канал (нет
процесса — нет доступа), а физический процесс продолжает работать на PLC — это
безопасное направление. Запускайте otgate под супервизором (systemd, restart-
политика контейнера) и направьте мониторинг на `/health`.

## Производительность и точность сигнала

otgate — это слой политики, а не преобразователь сигнала: он решает, пропустить ли
вызов, но не меняет само значение. Измерено на встроенном asyncua-симуляторе
реактора через реальное `opc.tcp://` соединение:

- **Нет искажения сигнала.** Значения, записанные через otgate, читаются с сервера
  бит-в-бит идентично — максимальная абсолютная ошибка `0` на значениях от `0.1`
  до `123456.789` и `π` с полной двойной точностью. otgate ничего не округляет,
  не масштабирует и не приводит; значение уходит в `asyncua` как есть, тип держит
  сервер.
- **Нет потери данных** на разрешённом трафике; **осознанная** потеря на
  запрещённом — запись с `DENY`/`ASK` никогда не доходит до сервера (в этом и
  смысл), а каждый вызов, разрешённый или нет, даёт ровно одну append-only строку
  аудита.
- **Overhead по задержке ≈ +0.16 мс/вызов** (в среднем) поверх сырого чтения
  бэкенда — около **+20 %** относительно субмиллисекундного локального чтения.
  Сама проверка политики — микросекунды; `audit.record()` ~0.03 мс (файловый
  хендл держится открытым и сбрасывается, а не переоткрывается на каждый вызов).
  Запись добавляет одно лишнее чтение бэкенда для проверок rate/interlock
  (осознанный round-trip ради безопасности).

Это с запасом быстро для назначения — LLM/агент выставляет уставки в масштабе
секунд. **otgate не предназначен стоять внутри realtime-контура управления**
(scan-цикл PLC, миллисекундный детерминизм); детерминированное управление
остаётся ниже него, в PLC/DCS. Долговечность аудита — на уровне ОС (сброс на
каждой записи, переживает падение процесса), но без `fsync` на каждую строку —
более строгий режим в TODO.

## Тесты

```bash
pytest -m "not integration"   # юнит-тесты — без зависимости от OPC UA (обещание по умолчанию)
pytest -m integration         # end-to-end против реального asyncua-сервера (нужен extra opcua)
pytest                         # всё
```

**Юнит**-набор работает целиком на FakeBackend без внешних зависимостей. Каждое правило политики покрыто тестом: в диапазоне → ALLOW/ASK, вне диапазона → DENY, ESD interlock → DENY, превышение скорости → DENY, тег с подтверждением → ASK, read-only → DENY, тег не из политики → DENY — плюс валидация политик, поля аудита, shadow mode, персистентность rate-истории, workflow подтверждения, auth/изоляция каналов, per-agent политики и поведение fail-closed/health.

**Интеграционный** набор (`tests/test_integration_asyncua.py`) поднимает asyncua-симулятор реактора на свободном порту и прогоняет те же сценарии — включая подтверждение, реально попадающее на сервер, и остановку сервера на лету, которая обязана fail-closed — по живому `opc.tcp://` соединению. Пропускается автоматически, если `asyncua` не установлен.

Оба набора гоняются в CI (GitHub Actions) на Python 3.11 и 3.12.

## Чего otgate пока НЕ делает (v0.1)

Честные ограничения — это осознанные сокращения, а не упущения:

- **Нет ролей агентов.** Все вызывающие равны; политики по идентичности нет.
- **Нет временных окон.** Нельзя выразить «только в рабочую смену».
- **Только скаляры.** Значения — числа или булевы; массивы/структуры не поддерживаются.
- **Один клиент.** Нет многопользовательского доступа, очередей и блокировок.
- **Конфигурация агентов статична.** Несколько агентов, каждый со своим токеном, политикой и идентичностью в аудите, поддерживаются через `agents.yaml`. Но набор агентов читается при старте — нет живого добавления/отзыва, изменение требует перезапуска. Токены долгоживущие (без ротации/истечения).
- **`max_rate` — по wall-clock и в пределах процесса.** Скорость считается относительно последней записи, которую otgate пропустил, а не по реальному тренду из историана. По умолчанию история хранится in-memory; задайте `OTGATE_RATE_HISTORY` (путь к файлу), чтобы она переживала перезапуск (иначе рестарт обнуляет её, и первая запись в rate-ограниченный тег после перезапуска не проверяется на скорость).
- **Нет GUI.**
